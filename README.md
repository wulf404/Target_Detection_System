# Target Detection System

Система обнаружения, сопровождения и передачи координат цели для башенной платформы.
Приложение написано на Qt/C++ и рассчитано на Linux/Jetson-класс устройства с SocketCAN,
USB-камерой, CUDA/OpenCV DNN, дальномером по UART и внешним источником координат.

## Что Делает Система

Система объединяет четыре потока данных:

```text
USB camera -> YOLO -> TargetManager -> AutoTracker -> CAN 0x05
External UART angles -> RemoteTracker -> TargetManager -> CAN 0x05
Rangefinder UART -> RangefinderUart -> CAN 0x09
CAN telemetry -> can_work -> TurretState / UI console
```

Приоритет источников такой:

1. Основная камера с YOLO, если цель свежая.
2. Внешний источник координат, если камера потеряла цель.
3. Режим `None`, если оба источника устарели или невалидны.

## Основные Модули

| Файл | Назначение |
| --- | --- |
| `main.cpp` | Запуск Qt-приложения, регистрация Qt metatype, подготовка CAN-окружения. |
| `mainwindow.*` | UI, запуск сервисов, отображение видео, дальномер, CAN-консоль. |
| `camera.*` | Захват USB-видео через GStreamer/OpenCV, реконнект камеры, вызов YOLO. |
| `my_yolo.*` | Загрузка ONNX-модели, инференс, NMS, выбор основной цели. |
| `target_manager.*` | Выбор активного источника цели, таймауты, watchdog. |
| `auto_tracker.*` | Расчет поправки по пикселям, фильтр, rate-limit команд сопровождения. |
| `turret_command.*` | Кодирование углов в CAN-кадр `0x05`. |
| `can_work.*` | SocketCAN RX/TX сервис, очередь отправки, парсинг телеметрии. |
| `tower_state.*` | Последние углы башни и признак свежести телеметрии. |
| `rangefinder_uart.*` | UART-протокол дальномера и отправка дальности в CAN `0x09`. |
| `uart_receiver.*` | Внешний UART-источник координат `az/el * 100`. |
| `serial_port_resolver.*` | Автовыбор USB-UART портов по driver/product/VID/PID. |
| `cuda_preprocess.*` | CUDA-препроцессинг кадра перед OpenCV DNN. |

## Отказоустойчивость

В текущей версии добавлены базовые защитные механизмы:

- CAN-запись идет через единую очередь `can_work`, а не напрямую из разных потоков.
- CAN-прием можно отключать отдельно от CAN-сервиса отправки.
- SocketCAN проверяет ошибки `socket`, `ioctl`, `bind`, `poll`, `read`, `write`.
- Watchdog `TargetManager::refresh()` регулярно сбрасывает устаревшие источники.
- Автосопровождение не отправляет команды без свежей телеметрии башни.
- При уходе с камеры на другой источник фильтр `AutoTracker` сбрасывается.
- Внешние координаты отбрасываются, если выходят за допустимый диапазон.
- UI больше не отправляет неинициализированный CAN-кадр при пустом выборе команды.

## Зависимости

Проект ожидает окружение примерно такого класса:

- Linux с SocketCAN.
- Qt 5 Widgets + SerialPort.
- OpenCV с CUDA/DNN модулями.
- CUDA/cuDNN.
- GStreamer с `v4l2src`, `jpegparse`, `jpegdec`, `videoconvert`, `appsink`.
- Компилятор `aarch64-linux-gnu-g++` для Jetson Orin NX.

Ключевые настройки лежат в `SO_rev1.pro`:

```qmake
CONFIG += c++17
DEFINES += ENABLE_CUDA_PREPROCESS=1
DEFINES += OPENCV_CPU_THREADS=1
CUDA_ARCH = sm_87
```

## Сборка

На целевом устройстве:

```bash
qmake SO_rev1.pro
make -j$(nproc)
```

Если сборка падает на CUDA/OpenCV, сначала проверьте пути в `SO_rev1.pro`:

- `/usr/local/cuda`
- `/usr/local/include/opencv4`
- `/usr/include/opencv4`
- `/usr/local/lib`
- `/usr/bin/aarch64-linux-gnu-g++`

## Запуск

Перед запуском должны быть доступны:

- CAN-интерфейс, обычно `can0`.
- Камера `/dev/video0`.
- ONNX-модель и файл классов.
- UART дальномера FTDI/FT232.
- UART внешнего источника CP210x, если он используется.

Приложение при старте:

1. Выполняет скрипт подготовки CAN из `main.cpp`.
2. Открывает главное окно Qt.
3. Открывает отдельное окно видео `YOLO Video`.
4. Автоматически ищет USB-UART устройства.
5. Ждет появления CAN-интерфейса и запускает CAN-сервис.
6. Автоматически стартует видеообработку.

## Протоколы

### CAN `0x05`: положение башни

Команда формируется в `turret_command.cpp`.

```text
byte 0: control
byte 1: azimuth high byte, centideg signed
byte 2: azimuth low byte
byte 3: elevation high byte, centideg signed
byte 4: elevation low byte
```

### CAN `0x09`: дальность

Дальность формируется в `MainWindow::showDistance`.
Сейчас дальномер выдает значение в миллиметрах внутри приложения, а в CAN отправляется масштабированное значение `mm / 100`.
Это место стоит сверить с документацией принимающего устройства.

### Внешний UART

`uart_receiver.cpp` сейчас читает 4 байта:

```text
azimuth_int16_be
elevation_int16_be
```

Значения интерпретируются как градусы, умноженные на 100.
Пара `-1, -1` означает отсутствие валидной цели.

Для промышленной надежности лучше расширить протокол до:

```text
header + version + sequence + payload + crc
```

## Диагностика

Полезные признаки в консоли приложения:

| Сообщение | Значение |
| --- | --- |
| `[CAN] Interface detected` | CAN-интерфейс найден, сервис запущен. |
| `[CAN] TX failed` | Ошибка отправки в SocketCAN. |
| `[TARGET] source=camera` | Активный источник цели - камера. |
| `[TARGET] source=external` | Активный источник цели - внешний UART. |
| `[TARGET] source=none` | Нет свежей цели. |
| `[RF] Opened rangefinder` | Дальномер найден и открыт. |
| `[UART] Opened coordinate source` | Внешний источник координат найден и открыт. |
| `[AUTO TRACK]` | Автосопровождение отправило новую позицию. |

## Безопасная Эксплуатация

Перед работой на реальном стенде проверьте:

- CAN-команды уходят только после свежей телеметрии башни.
- Лимиты углов соответствуют механическим ограничениям конкретной платформы.
- При отключении камеры источник меняется ожидаемо.
- При отключении внешнего UART система переходит в `source=none`.
- При отключении CAN нет накопления опасной очереди старых команд.
- Дальность в CAN `0x09` соответствует ожидаемым единицам принимающего устройства.
